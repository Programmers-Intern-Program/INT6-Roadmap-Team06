# AI 개발자 성장 코치 서비스 사용자 흐름 및 화면 설계 정렬본

## 1. 문서 목적

이 문서는 기존 사용자 흐름 및 화면 설계를 제안서, 아키텍처 문서, 이후 계약 문서 기준에 맞춰 정렬한 버전이다.

핵심 목적
- 화면 입력값과 API 계약의 불일치 제거
- 상태 표현과 저장 규칙 차이 제거
- GitHub 중심 v1 흐름과 v2 대화형 확장 흐름을 같은 기준으로 정렬
- 로드맵 원본 구조와 진도 계산 기준 반영

적용 기준
- 화면 입력 필드는 API 명세 정렬본을 따른다
- enum 표시값은 프론트에서 한글 라벨로 매핑하고 저장/전송 값은 enum 원문을 사용한다
- 로드맵 원본은 `learning_roadmaps + roadmap_weeks + progress_logs`다
- 화면의 진도 상태는 `progress_logs` 최신 row 기준으로 계산한다

---

## 2. 설계 기준

- v1은 페이지 기반 GitHub 중심 사용자 흐름을 따른다
- v2는 v1 결과를 재사용하는 AI 코치 채팅 흐름을 추가한다
- 각 화면은 입력, 결과 확인, 다음 단계 이동이 명확해야 한다
- 로드맵 화면은 표시용 `weekNumber`와 저장용 `roadmapWeekId`를 함께 다뤄야 한다
- 인증 후 사용자 식별은 JWT 컨텍스트에서 처리하며 화면은 `userId`를 직접 다루지 않는다

---

## 3. 전체 사용자 흐름

### 3.1 v1 전체 흐름

1. 사용자가 로그인한다
2. 사용자가 목표 직무, 현재 수준, 기술 스택, 선택 자료를 입력하고 프로필을 저장한다
3. 사용자가 GitHub OAuth 연동을 완료한다
4. 사용자가 분석 대상 저장소와 핵심 repo를 선택한다
5. 시스템이 전체 GitHub 정적 분석과 핵심 repo 요약을 수행한다
6. 사용자가 GitHub 분석 결과를 검토하고 필요 시 보정한다
7. 시스템이 프로필과 GitHub 최종 분석 결과를 기준으로 역량 진단을 수행한다
8. 시스템이 역량 진단 결과를 바탕으로 학습 로드맵을 생성한다
9. 사용자가 로드맵을 확인하고 주차별 진도 체크를 진행한다
10. 사용자가 대시보드에서 최근 결과를 다시 조회한다

### 3.2 v2 확장 흐름

1. 사용자가 AI 코치 채팅방에 진입한다
2. 시스템이 먼저 코치 세션을 생성한다
3. 세션 생성 시점의 `profileVersion`, `roadmapVersion`을 고정한다
4. Coach가 Tier 1, Tier 2, Tier 3 컨텍스트를 불러온다
5. 사용자가 메시지를 보내면 시스템이 응답과 의도를 반환한다
6. 필요 시 스트리밍 응답을 SSE로 수신한다
7. 필요 시 재분석 또는 재계획 이벤트가 발생한다
8. 새 로드맵이 생성되더라도 진행 중인 세션은 자동으로 버전을 바꾸지 않고, 다음 대화 또는 사용자 확인 후 반영한다

## 4. 화면 목록

## 4.1 로그인 / 회원가입 화면

목적
- 서비스 진입과 계정 인증

핵심 요소
- 이메일 로그인
- OAuth 로그인
- 회원가입
- 비밀번호 찾기

계약 기준
- 인증 성공 후 JWT 기반 보호 API 접근
- 화면은 로그인 제공자 차이를 숨기되, 서버 저장은 `auth_provider` 기준으로 관리

---

## 4.2 프로필 입력 / 수정 화면

목적
- 진단과 로드맵에 필요한 기본 입력값을 수집하고 저장한다

핵심 요소
- 목표 직무 선택
- 현재 수준 선택
- 기술 스택 입력
- 관심 분야 입력
- 주당 학습 가능 시간 입력
- 목표 날짜 입력
- 이력서 자료 선택
- 포트폴리오 자료 선택
- 저장 버튼
- GitHub 연동 시작 버튼

입력값
- `targetRole`
- `currentLevel`
- `skills[]`
  - `skillName`
  - `proficiencyLevel`
- `interestAreas[]`
- `weeklyStudyHours`
- `targetDate`
- `resumeAssetId(optional)`
- `portfolioAssetId(optional)`

입력 규칙
- `targetRole`은 자유 텍스트가 아니라 사전에 정의된 직무 코드 선택이다
- `currentLevel`, `proficiencyLevel`은 사전 정의 enum 선택이다
- `weeklyStudyHours`는 1~40 범위를 사용한다
- `targetDate`는 오늘 이후 날짜만 허용한다
- 자료 선택은 선택값이며, 없는 경우에도 다음 단계로 이동할 수 있다

행동
- 저장 시 프로필 저장 API 호출
- 저장 완료 후 GitHub 연동 화면으로 이동

---

## 4.3 GitHub 연동 및 저장소 선택 화면

목적
- GitHub 계정을 연결하고 분석할 저장소와 핵심 repo를 고른다

핵심 요소
- GitHub OAuth 연동 버튼
- 연결 상태 표시
- 저장소 목록
- 분석 대상 저장소 선택
- 핵심 repo 선택
- 분석 시작 버튼

입력값
- `githubConnectionId`
- `selectedRepositoryIds[]`
- `coreRepositoryIds[]`

상태
- 연결 전
- 연결 완료
- 저장소 불러오는 중
- 저장소 선택 완료
- 분석 시작 가능

계약 기준
- GitHub는 URL 수기 입력이 아니라 OAuth 연결 기준으로 처리한다
- `selectedRepositoryIds[]`는 전체 정적 분석 대상이다
- `coreRepositoryIds[]`는 LLM 요약 대상이다
- 핵심 repo는 분석 대상 저장소의 부분집합이어야 한다

예외 처리
- OAuth 실패 안내
- 권한 부족 안내
- 저장소 조회 실패 시 재시도 버튼 제공

---

## 4.4 GitHub 분석 결과 및 사용자 보정 화면

목적
- 실제 사용 기술 분석 결과를 보여주고, 사용자가 자신의 경험에 맞게 보정한다

핵심 요소
- 정적 분석 요약
- 핵심 repo 요약 카드
- 기술 태그 목록
- 사용 깊이 후보 표시
- 근거 정보 표시
- 사용자 보정 입력
- 최종 기술 프로필 저장 버튼
- 역량 진단 시작 버튼

표시 데이터
- `staticSignals`
- `repoSummaries[]`
- `techTags[]`
- `depthEstimates[]`
- `evidences[]`
- `userCorrections[]`
- `finalTechProfile`

상태
- 분석 중
- 분석 완료
- 보정 전
- 보정 저장 완료

계약 기준
- AI는 후보와 근거를 제시하고, 사용자가 이를 수정할 수 있어야 한다
- 사용자 보정은 AI 추정값과 구분해 저장되어야 한다
- 최종 진단은 보정된 GitHub 분석 결과를 기준으로 수행한다

예외 처리
- 분석 실패 시 재시도 버튼 제공
- 보정 저장 실패 시 임시 편집값 유지

---

## 4.5 역량 진단 결과 화면

목적
- GitHub 최종 분석이 반영된 역량 진단 결과와 부족 기술 요약을 제공한다

핵심 요소
- 현재 역량 요약 카드
- 부족 기술 목록
- 강점 요약
- 우선순위 추천
- 추천 근거
- GitHub 반영 요약
- 로드맵 생성 버튼

표시 데이터
- `summary`
- `missingSkills[]`
- `strengths[]`
- `recommendations[]`
- `version`

행동
- 로드맵 생성 화면으로 이동
- 이전 GitHub 분석 결과 상세 보기

계약 기준
- 화면은 최신 결과를 기본으로 보여주되, 상세 조회는 `diagnosisId` 기준으로 고정한다
- severity 표시는 프론트 한글 라벨로 변환해 보여준다

---

## 4.6 학습 로드맵 생성 화면

목적
- 역량 진단 결과를 바탕으로 로드맵을 생성한다

핵심 요소
- 진단 결과 선택
- 주당 학습 가능 시간 override 입력
- 목표 날짜 override 입력
- 로드맵 생성 버튼

입력값
- `diagnosisId`
- `weeklyStudyHours`
- `targetDate`

계약 기준
- `diagnosisId`는 필수다
- `weeklyStudyHours`, `targetDate`는 프로필 값을 덮어쓰는 override 용도다
- v1 생성 입력은 진단 결과 중심으로 유지하고, GitHub 분석 결과는 진단 단계에서 흡수된 것으로 본다
- 생성된 결과에는 `roadmapId`, `version`, `weeks[]`가 포함된다

---

## 4.7 학습 로드맵 조회 / 진도 관리 화면

목적
- 주차별 학습 계획과 추천 자료를 제공하고 진도 상태를 저장한다

핵심 요소
- 주차별 로드맵 요약
- 학습 주제
- 추천 자료
- 추천 이유
- 예상 학습 시간
- 진도 상태 버튼
- 메모 입력
- 완료 표시 버튼

표시/저장 규칙
- 화면에는 `weekNumber`를 표시한다
- 내부 저장 시에는 `roadmapWeekId`를 사용한다
- 화면의 `progressStatus`, `progressNote`, `progressUpdatedAt`는 `progress_logs` 최신 row 기준으로 계산한다
- `roadmap_payload`만 보고 진도 상태를 계산하지 않는다

행동
- 주차별 상세 펼치기
- 상태 저장
- 메모 저장

상태 규칙
- 상태 이력은 append-only다
- `DONE -> TODO` 직접 복귀는 허용하지 않는다

---

## 4.8 마이페이지 또는 대시보드

목적
- 현재까지의 결과를 한눈에 확인하고 최신 결과로 이동한다

핵심 요소
- 최근 프로필 요약
- 최근 GitHub 최종 분석 요약
- 최근 진단 결과 요약
- 최근 로드맵 진행률
- 각 상세 화면 바로가기

계약 기준
- 기본 카드 값은 최신 version 기준으로 보여준다
- 상세 이동은 각 결과의 ID 기준으로 고정한다

## 5. v2 추가 화면

## 5.1 AI 코치 채팅 화면

목적
- 대화형 학습 코칭을 제공한다

핵심 요소
- 채팅 메시지 목록
- 입력창
- 오늘의 할 일 카드
- 진도 요약
- 재계획 제안 메시지
- 스트리밍 응답 영역

행동
- 채팅 진입 시 먼저 세션 생성 API 호출
- `sessionId` 발급 후 메시지 전송 API 호출
- 필요 시 SSE 기반 응답 수신
- 재계획 확인 요청 처리

계약 기준
- 세션 시작 시 `profileVersion`, `roadmapVersion` 고정
- Coach는 기본적으로 Tier 1, Tier 2, Tier 3을 읽는다
- 현재 세션 중 새 결과가 생성되어도 화면은 자동으로 기준 버전을 바꾸지 않는다
- `detectedIntent`는 내부 enum을 프론트 라벨로 변환해 표시한다

## 5.2 코딩테스트 분석 화면 draft

목적
- v2 후순위 확장으로 코딩테스트 약점 분석 화면을 추가할 수 있다

핵심 요소
- 제출 이력 연결 또는 업로드
- 취약 유형 요약
- 반복 실패 패턴 표시
- 추천 문제 또는 복습 제안

계약 기준
- 이 화면은 v1 필수 흐름에 포함되지 않는다
- 입력 형식과 분석 범위는 후속 계약 배치에서 별도로 닫는다

## 6. 화면 간 이동 구조

- 로그인 → 프로필 입력/수정
- 프로필 입력/수정 → GitHub 연동 및 저장소 선택
- GitHub 연동 및 저장소 선택 → GitHub 분석 결과 및 사용자 보정
- GitHub 분석 결과 및 사용자 보정 → 역량 진단 결과
- 역량 진단 결과 → 학습 로드맵 생성
- 학습 로드맵 생성 → 학습 로드맵 조회/진도 관리
- 학습 로드맵 조회/진도 관리 → 대시보드
- 대시보드 → AI 코치 채팅(v2)

## 8. 설계 원칙 요약

- 각 화면은 다음 단계 행동이 명확해야 한다
- 화면 입력값은 API 필드명과 의미를 맞춘다
- enum 저장값과 한글 표시값을 분리한다
- v1은 페이지 중심, v2는 대화형 확장 구조를 따른다
- 로드맵 원본은 `learning_roadmaps + roadmap_weeks + progress_logs`다
- 진도 상태는 `progress_logs` 최신 row 기준으로 계산한다
- v2 내부 구조가 있더라도 v1 화면은 단순하고 일관된 사용 흐름을 유지해야 한다

## 9. 이번 정렬본에서 핵심 수정한 부분

- 프로필 입력 이후 GitHub OAuth 연동과 저장소 선택 단계를 명시적으로 분리
- GitHub 입력 계약을 `githubConnectionId`, `selectedRepositoryIds[]`, `coreRepositoryIds[]` 기준으로 정리
- GitHub 분석 결과 화면에 사용자 보정과 최종 기술 프로필 단계를 추가
- 로드맵 생성 입력을 `diagnosisId` 중심으로 단순화
- 코딩테스트 화면을 v1 본 흐름에서 제거하고 v2 draft로 분리
- 코치 화면에 `profileVersion`, `roadmapVersion` 고정 규칙 반영
