# AI 개발자 성장 코치 서비스 사용자 흐름 및 화면 설계 정렬본

## 1. 문서 목적

이 문서는 기존 사용자 흐름 및 화면 설계를 DB 물리 스키마 문서, 계약 부록 문서, API 명세 정렬본 기준에 맞춰 정렬한 버전이다.

핵심 목적
- 화면 입력값과 API 계약의 불일치 제거
- 상태 표현과 저장 규칙 차이 제거
- 로드맵 원본 구조와 진도 계산 기준 반영
- v1 페이지 흐름과 v2 대화형 확장 흐름을 같은 계약 위에 정렬

적용 기준
- 화면 입력 필드는 API 명세 정렬본을 따른다
- enum 표시값은 프론트에서 한글 라벨로 매핑하고 저장/전송 값은 enum 원문을 사용한다
- 로드맵 원본은 `learning_roadmaps + roadmap_weeks + progress_logs`다
- 화면의 진도 상태는 `progress_logs` 최신 row 기준으로 계산한다

---

## 2. 설계 기준

- v1은 페이지 기반 사용자 흐름을 중심으로 설계한다
- v2는 v1 결과를 재사용하는 AI 코치 채팅 흐름을 추가한다
- 각 화면은 입력, 결과 확인, 다음 단계 이동이 명확해야 한다
- 로드맵 화면은 표시용 `weekNumber`와 저장용 `roadmapWeekId`를 함께 다뤄야 한다
- 인증 후 사용자 식별은 JWT 컨텍스트에서 처리하며 화면은 `userId`를 직접 다루지 않는다

## 3. 전체 사용자 흐름

### 3.1 v1 전체 흐름

1. 사용자가 로그인한다
2. 사용자가 목표 직무, 현재 수준, 기술 스택을 입력하고 프로필을 저장한다
3. 시스템이 프로필 기준으로 1차 역량 진단을 수행한다
4. 사용자가 GitHub URL을 입력한다
5. 시스템이 실제 사용 기술을 분석하고 진단 결과를 보정한다
6. 사용자가 코딩테스트 풀이 이력을 입력 또는 업로드한다
7. 시스템이 코테 약점을 분석한다
8. 시스템이 진단 결과와 약점 결과를 바탕으로 학습 로드맵을 생성한다
9. 사용자가 로드맵을 확인하고 주차별 진도 체크를 진행한다
10. 사용자가 대시보드에서 최근 결과를 다시 조회한다

### 3.2 v2 확장 흐름

1. 사용자가 AI 코치 채팅방에 진입한다
2. 시스템이 먼저 코치 세션을 생성한다
3. 세션 생성 시점의 `profileVersion`, `planVersion`을 고정한다
4. Coach가 최신 진단 결과, GitHub 분석 결과, 코테 약점, 로드맵을 컨텍스트로 불러온다
5. 사용자가 메시지를 보내면 시스템이 응답과 의도를 반환한다
6. 필요 시 스트리밍 응답을 SSE로 수신한다
7. 필요 시 재분석 또는 재계획 이벤트가 발생한다

## 4. 화면 목록

## 4.1 로그인 / 회원가입 화면

목적
- 서비스 진입과 계정 인증

핵심 요소
- 이메일 로그인
- OAuth 로그인
- 회원가입
- 비밀번호 찾기

계약 기준
- 인증 성공 후 JWT 기반 보호 API 접근
- 화면은 로그인 제공자 차이를 숨기되, 서버 저장은 `auth_provider` 기준으로 관리

---

## 4.2 프로필 입력 / 수정 화면

목적
- 진단에 필요한 기본 입력값을 수집하고 저장한다

핵심 요소
- 목표 직무 선택
- 현재 수준 선택
- 기술 스택 입력
- 관심 분야 입력
- GitHub URL 입력
- 주당 학습 가능 시간 입력
- 목표 날짜 입력
- 저장 버튼
- 진단 시작 버튼

입력값
- `targetRole`
- `currentLevel`
- `skills[]`
  - `skillName`
  - `proficiencyLevel`
- `interestAreas[]`
- `githubUrl`
- `weeklyStudyHours`
- `targetDate`

입력 규칙
- `targetRole`은 자유 텍스트가 아니라 사전에 정의된 직무 코드 선택이다
- `currentLevel`, `proficiencyLevel`은 사전 정의 enum 선택이다
- `githubUrl`은 선택값이며 GitHub URL 형식만 허용한다
- `weeklyStudyHours`는 1~40 범위를 사용한다
- `targetDate`는 오늘 이후 날짜만 허용한다

행동
- 저장 시 프로필 저장 API 호출
- 진단 시작 시 저장된 `profileId` 기준으로 역량 진단 API 호출

---

## 4.3 역량 진단 결과 화면

목적
- 1차 역량 진단 결과와 부족 기술 요약을 제공한다

핵심 요소
- 현재 역량 요약 카드
- 부족 기술 목록
- 심각도 표시
- 우선순위 추천
- 추천 근거
- GitHub 분석으로 보정하기 버튼
- 코테 분석으로 이동 버튼
- 로드맵 생성 버튼

표시 데이터
- `summary`
- `missingSkills[]`
- `strengths[]`
- `recommendations[]`
- `version`

행동
- GitHub 분석 화면으로 이동
- 코테 약점 분석 화면으로 이동
- 로드맵 생성 화면으로 이동

계약 기준
- 화면은 최신 결과를 기본으로 보여주되, 상세 조회는 `diagnosisId` 기준으로 고정한다
- severity 표시는 프론트 한글 라벨로 변환해 보여준다

---

## 4.4 GitHub 연결 및 분석 화면

목적
- 실제 사용 기술 분석과 진단 보정을 수행한다

핵심 요소
- GitHub URL 입력
- 분석 시작 버튼
- 저장소 목록
- 실제 사용 기술 스택 요약
- 프로젝트별 근거 정보
- 자기기입 대비 차이점 표시
- 보정된 진단 결과 요약

입력값
- `githubUrl`
- `diagnosisId` 선택

상태
- 입력 전
- 분석 중
- 분석 완료
- 분석 실패

계약 기준
- 외부 계약상 입력은 `githubUrl`로 통일한다
- v1은 기본 동기 처리지만, 화면은 로딩 상태를 가진다
- 근거 정보의 타입은 enum 기준으로 정리해 표시한다

예외 처리
- 유효하지 않은 URL 안내
- 접근 불가 저장소 안내
- 분석 실패 시 재시도 버튼 제공

---

## 4.5 코딩테스트 이력 업로드 화면

목적
- 분석 대상이 되는 제출 이력을 입력하거나 업로드한다

핵심 요소
- 파일 업로드
- 직접 입력 폼
- 샘플 형식 안내
- 업로드 결과 요약

입력값
- `submissions[]` 또는 파일

상태
- 업로드 전
- 검증 중
- 업로드 완료
- 형식 오류

계약 기준
- 업로드 성공 시 `uploadId`를 반환받는다
- 이후 분석 화면에서 이 `uploadId`를 재사용할 수 있다
- 화면은 `userId`를 다루지 않는다

예외 처리
- 허용 형식 안내
- 크기 초과 안내
- 필수 컬럼 누락 안내

---

## 4.6 코딩테스트 약점 분석 화면

목적
- 유형별 취약점 분석과 추천 문제를 제공한다

핵심 요소
- 분석 범위 선택
  - 최근 업로드 기준
  - 전체 이력 기준
- 유형별 정답률
- 평균 풀이 시간
- 재시도 횟수
- 취약 유형 요약
- 추천 문제 목록

입력값
- `uploadId` 선택

상태
- 분석 전
- 분석 중
- 분석 완료
- 분석 실패

계약 기준
- `uploadId`를 보내면 해당 배치만 분석한다
- `uploadId`를 보내지 않으면 전체 제출 이력을 분석한다
- `problemType`과 `difficulty` 표시는 프론트 한글 라벨로 변환해 보여준다

---

## 4.7 학습 로드맵 생성 화면

목적
- 진단 결과와 분석 결과를 바탕으로 로드맵을 생성한다

핵심 요소
- 진단 결과 선택
- GitHub 분석 결과 선택
- 코테 분석 결과 선택
- 주당 학습 가능 시간 override 입력
- 목표 날짜 override 입력
- 로드맵 생성 버튼

입력값
- `diagnosisId`
- `githubAnalysisId` 선택
- `codingTestAnalysisId` 선택
- `weeklyStudyHours` 선택
- `targetDate` 선택

계약 기준
- `diagnosisId`는 필수다
- `weeklyStudyHours`, `targetDate`는 프로필 값을 덮어쓰는 override 용도다
- 생성된 결과에는 `roadmapId`, `version`, `weeks[]`가 포함된다

---

## 4.8 학습 로드맵 조회 / 진도 관리 화면

목적
- 주차별 학습 계획과 추천 자료를 제공하고 진도 상태를 저장한다

핵심 요소
- 12주 로드맵 요약
- 주차별 학습 주제
- 추천 강의, 문제, 프로젝트
- 예상 학습 시간
- 진도 상태 버튼
- 메모 입력
- 완료 표시 버튼

표시/저장 규칙
- 화면에는 `weekNumber`를 표시한다
- 내부 저장 시에는 `roadmapWeekId`를 사용한다
- 화면의 `progressStatus`, `progressNote`, `progressUpdatedAt`는 `progress_logs` 최신 row 기준으로 계산한다
- `roadmap_payload`만 보고 진도 상태를 계산하지 않는다

행동
- 주차별 상세 펼치기
- 상태 저장
- 메모 저장

상태 규칙
- 상태 이력은 append-only다
- `DONE -> TODO` 직접 복귀는 허용하지 않는다

## 4.9 마이페이지 또는 대시보드

목적
- 현재까지의 결과를 한눈에 확인하고 최신 결과로 이동한다

핵심 요소
- 최근 프로필 요약
- 최근 진단 결과 요약
- 최근 GitHub 분석 요약
- 최근 코테 약점 요약
- 최근 로드맵 진행률
- 각 상세 화면 바로가기

계약 기준
- 기본 카드 값은 최신 version 기준으로 보여준다
- 상세 이동은 각 결과의 ID 기준으로 고정한다

## 5. v2 추가 화면

## 5.1 AI 코치 채팅 화면

목적
- 대화형 학습 코칭을 제공한다

핵심 요소
- 채팅 메시지 목록
- 입력창
- 오늘의 할 일 카드
- 진도 요약
- 재계획 제안 메시지
- 스트리밍 응답 영역

행동
- 채팅 진입 시 먼저 세션 생성 API 호출
- `sessionId` 발급 후 메시지 전송 API 호출
- 필요 시 SSE 기반 응답 수신
- 재계획 확인 요청 처리

계약 기준
- 세션 시작 시 `profileVersion`, `planVersion` 고정
- 현재 세션 중 새 결과가 생성되어도 화면은 자동으로 기준 버전을 바꾸지 않는다
- `detectedIntent`는 내부 enum을 프론트 라벨로 변환해 표시한다

## 6. 화면 간 이동 구조

- 로그인 → 프로필 입력/수정
- 프로필 입력/수정 → 역량 진단 결과
- 역량 진단 결과 → GitHub 분석
- 역량 진단 결과 → 코테 이력 업로드
- 코테 이력 업로드 → 코테 약점 분석
- 역량 진단 결과 / GitHub 분석 / 코테 약점 분석 → 학습 로드맵 생성
- 학습 로드맵 생성 → 학습 로드맵 조회/진도 관리
- 학습 로드맵 조회/진도 관리 → 대시보드
- 대시보드 → AI 코치 채팅(v2)

## 8. 설계 원칙 요약

- 각 화면은 다음 단계 행동이 명확해야 한다
- 화면 입력값은 API 필드명과 의미를 맞춘다
- enum 저장값과 한글 표시값을 분리한다
- v1은 페이지 중심, v2는 대화형 확장 구조를 따른다
- 로드맵 원본은 `learning_roadmaps + roadmap_weeks + progress_logs`다
- 진도 상태는 `progress_logs` 최신 row 기준으로 계산한다
- v2 내부 구조가 있더라도 v1 화면은 단순하고 일관된 사용 흐름을 유지해야 한다

## 9. 이번 정렬본에서 핵심 수정한 부분

- 프로필 입력 화면을 별도 단계로 분리
- GitHub 입력 계약을 `githubUrl` 기준으로 통일
- 코테 분석 범위를 `uploadId` 유무 기준으로 명확화
- 로드맵 진도 계산 기준을 `progress_logs`로 명확화
- 화면이 `userId`를 직접 다루지 않도록 정리
- v2 코치 화면에 세션 생성 선행 규칙 반영
