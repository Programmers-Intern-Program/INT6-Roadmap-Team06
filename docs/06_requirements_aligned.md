# AI 개발자 성장 코치 서비스 요구사항 정렬본

## 1. 문서 목적

이 문서는 기존 요구사항 문서를 제안서, 아키텍처 문서, 이후 계약 문서 기준에 맞춰 팀 개발 계약 수준으로 정렬한 버전이다.

핵심 목적
- 상위 요구사항과 하위 계약 문서 간 충돌 제거
- v1 MVP와 v2 확장 범위를 같은 기준으로 재정렬
- GitHub 중심 진단 흐름과 로드맵 생성 기준을 요구사항 수준에서 고정
- 팀원별 해석 차이를 줄이기 위한 공통 참조점 제공

적용 기준
- 저장 구조는 DB 물리 스키마 문서를 따른다
- enum, JSONB shape, validation, 상태 전이는 계약 부록 문서를 따른다
- 외부 인터페이스는 API 명세 정렬본을 따른다
- 기능별 상세 처리와 화면 동작은 기능 명세 정렬본, 사용자 흐름 및 화면 설계 정렬본을 따른다

---

## 2. 서비스 목표

- 사용자의 현재 역량을 진단한다
- 목표 직무 대비 부족한 기술과 우선순위를 도출한다
- GitHub 기반 실제 사용 기술을 분석해 진단 결과를 보정한다
- 학습 계획을 포트폴리오로 이어질 수 있는 행동 단위까지 제안한다
- 진단 결과를 바탕으로 주차별 학습 로드맵을 생성한다
- v2에서는 멀티 에이전트 기반 AI 코치로 학습 실행과 재계획을 지원한다

---

## 3. 핵심 사용자

- 백엔드 개발자 취업 준비생
- 부트캠프, 데브코스, 강의 수강생
- 프로젝트 경험은 있지만 기술 깊이를 정리하기 어려운 주니어 개발자
- GitHub 활동을 취업 준비와 포트폴리오 정리로 연결하고 싶은 사용자

---

## 4. 사용자 문제

- 현재 실력을 객관적으로 파악하기 어렵다
- 목표 직무 대비 어떤 기술이 부족한지 알기 어렵다
- GitHub 경험과 학습 계획, 취업 준비가 따로 관리된다
- 무엇을 먼저 공부해야 할지 우선순위를 잡기 어렵다
- 계획을 세워도 실천으로 이어지지 않는 경우가 많다
- 학습 경험을 포트폴리오와 프로젝트 기술서로 연결하기 어렵다

## 6. 기능 요구사항

### 6.1 인증 및 프로필

- 사용자는 이메일 로그인 또는 OAuth 로그인을 사용할 수 있어야 한다
- 보호 API는 JWT 기반 인증 컨텍스트를 사용해야 한다
- 사용자는 목표 직무, 현재 수준, 기술 스택, 관심 분야, 주당 학습 시간, 목표 날짜를 저장할 수 있어야 한다
- 사용자는 이력서 또는 포트폴리오 자료를 선택적으로 연결할 수 있어야 한다
- 사용자 식별용 `userId`는 요청 body가 아니라 인증 컨텍스트에서 처리해야 한다
- 목표 직무는 자유 텍스트가 아니라 사전 정의된 직무 코드 기준으로 선택해야 한다
- 현재 수준과 숙련도는 사전 정의 enum 값만 허용해야 한다

### 6.2 GitHub 연동 및 분석

- 사용자는 GitHub 계정명이나 URL 수기 입력이 아니라 OAuth 기반 연동을 통해 계정을 연결해야 한다
- 시스템은 사용자가 분석 대상으로 선택한 저장소 목록을 기준으로 메타데이터와 정적 신호를 수집해야 한다
- 사용자는 핵심 repo를 별도로 선택할 수 있어야 하며, 시스템은 이 저장소에 대해 LLM 기반 요약을 생성해야 한다
- 시스템은 실제 사용 기술 스택을 기술 태그와 사용 깊이 후보 형태로 추출해야 한다
- 시스템은 근거 정보와 함께 사용자가 수정 가능한 보정 구조를 제공해야 한다
- AI는 최종 확정자가 아니라 후보와 근거를 제시하는 역할이어야 한다
- GitHub 분석 결과는 별도 결과 이력으로 저장해야 한다

### 6.3 역량 진단

- 시스템은 저장된 프로필과 GitHub 최종 분석 결과를 기준으로 역량 진단을 수행해야 한다
- 시스템은 직무별 필요 역량 기준표와 비교해 부족 기술을 도출해야 한다
- 시스템은 현재 역량 요약, 부족 기술 목록, 우선순위, 추천 근거를 제공해야 한다
- 역량 진단 결과는 기존 결과를 수정하지 않고 새 결과 이력으로 저장해야 한다
- 기본 조회는 최신 결과를 반환할 수 있어야 하며, 개별 결과 조회도 가능해야 한다

### 6.4 학습 로드맵 생성

- 시스템은 역량 진단 결과를 바탕으로 로드맵을 생성해야 한다
- 시스템은 학습 우선순위와 선행 관계를 반영한 주차별 계획을 제공해야 한다
- 시스템은 공식 문서 읽기, 예제 구현, 블로그 정리, 미니 프로젝트 적용 같은 실행 가능한 작업 단위를 포함해야 한다
- 시스템은 각 주차의 추천 이유를 함께 제공해야 한다
- 사용자는 주차 단위 진도 체크를 수행할 수 있어야 한다
- 로드맵 원본 데이터는 `learning_roadmaps + roadmap_weeks + progress_logs` 구조로 유지해야 한다
- `roadmap_payload`는 결과 보관 및 snapshot 조립용이며 원본 데이터로 간주하지 않는다

### 6.5 결과 저장 및 재조회

- 사용자 입력값과 분석 결과는 저장되어야 한다
- 사용자는 이전 진단 결과, GitHub 분석 결과, 로드맵을 다시 조회할 수 있어야 한다
- 결과 이력 관리를 위해 `version` 필드를 사용해야 한다
- `version`은 서비스 단계인 v1/v2와 별개로, 개별 결과 이력 관리용으로 사용해야 한다
- v1 원본 데이터는 v2에서 재사용 가능해야 한다

### 6.6 v2 코치 확장

- AI 코치는 세션 생성 후 대화를 시작해야 한다
- 세션 생성 시점의 `profileVersion`, `roadmapVersion`은 고정돼야 한다
- AI 코치는 최신 진단 결과와 로드맵을 읽어 오늘의 할 일과 진도 점검을 제공해야 한다
- 필요 시 재분석 또는 재계획 이벤트를 발행할 수 있어야 한다
- 재계획 결과가 생성되어도 진행 중인 세션은 자동으로 새 버전으로 전환되지 않아야 한다

### 6.7 v2 Context / Event 기준

- Context Manager는 Tier 조립, 버전 관리, 캐싱, 압축 오케스트레이션을 담당해야 한다
- Context Manager는 비즈니스 판단, 직접적인 LLM 응답 생성, 이벤트 발행 책임을 가지지 않아야 한다
- Analyzer는 기본적으로 Tier 1을, Planner는 Tier 1과 Tier 2를, Coach는 Tier 1, Tier 2, Tier 3을 읽어야 한다
- 이벤트는 notification + pull 패턴을 따라야 하며, payload에는 최소 메타데이터만 담아야 한다
- 같은 사용자에 대한 재분석, 재계획은 쿨다운과 사용자 확인 기준을 둘 수 있어야 한다

## 7. 비기능 요구사항

### 7.1 성능

- 기본 조회, 진도 저장은 사용자가 기다릴 수 있는 수준의 시간 안에 응답해야 한다
- v1 외부 계약은 기본 동기 처리로 시작한다
- GitHub 분석, 핵심 repo 요약 등 장시간 작업은 추후 비동기 상태 확인 방식으로 확장 가능해야 한다

### 7.2 보안

- 사용자 인증은 JWT 또는 OAuth 기반으로 처리해야 한다
- GitHub 연동 토큰과 외부 인증 정보는 안전하게 저장하고 최소 범위만 사용해야 한다
- 민감 정보는 로그와 응답에서 노출되지 않아야 한다
- 사용자 소유가 아닌 결과 ID 접근은 차단돼야 한다

### 7.3 데이터 일관성

- 결과 row는 append-only 저장을 우선한다
- 로드맵 진도는 `progress_logs` 최신 row 기준으로 계산해야 한다
- enum, JSONB key, validation은 공통 계약 문서를 따라야 한다
- 같은 의미의 데이터가 문서마다 다른 이름이나 다른 shape로 표현되지 않아야 한다
- AI 추정값과 사용자 확정값은 구분해 저장할 수 있어야 한다

### 7.4 확장성

- v1은 단순 파이프라인 구조로 시작하되, v2 멀티 에이전트 확장이 가능해야 한다
- 컴포넌트 간 경계를 분명히 두어 구조 변경 비용을 낮춰야 한다
- 결과 저장 구조는 snapshot 조립이 가능하도록 설계해야 한다

### 7.5 관측 가능성

- 기본 로그와 예외 추적이 가능해야 한다
- v2에서는 에이전트별 호출 횟수, 토큰, 비용, 응답 시간 수집이 가능해야 한다

### 7.6 AI 신뢰성

- LLM 출력은 구조화된 결과로 검증 가능해야 한다
- GitHub 분석에서 AI는 후보와 근거를 제시하고, 사용자가 이를 보정할 수 있어야 한다
- Planner는 v2에서 주제 구조 생성과 검증된 자료 결합을 분리해 환각 가능성을 낮춰야 한다

## 8. 제외 범위

v1에서는 아래 기능을 필수 범위에서 제외한다.

- 실시간 AI 코치 채팅
- 완전한 멀티 에이전트 구조
- 고도화된 패턴 감지 자동 개입
- 코딩테스트 업로드 및 약점 분석
- 검증 자료 검색용 Function Calling의 세부 인터페이스
- 공개 API용 장기 작업 job 관리 API

## 9. 성공 기준

### 9.1 v1 성공 기준

- 사용자가 로그인 후 자신의 프로필과 선택 자료를 저장할 수 있다
- GitHub OAuth 연동 후 분석 저장소를 선택할 수 있다
- GitHub 기반 실제 사용 스택과 자기기입 스택 비교가 가능하다
- 사용자 보정이 반영된 GitHub 최종 분석 결과를 확인할 수 있다
- 부족 기술과 추천 근거를 확인할 수 있다
- 주차별 학습 로드맵을 생성하고 조회할 수 있다
- 진도 체크를 저장할 수 있다
- 결과가 이력으로 저장되고 다시 조회된다
- 로드맵 진도는 `progress_logs` 기준으로 일관되게 계산된다

### 9.2 v2 성공 기준

- AI 코치가 진단 결과와 로드맵을 바탕으로 대화할 수 있다
- Pattern Detector와 재계획이 이벤트 기반으로 연결된다
- 상태 관리와 결과 이력 관리가 분리된 구조로 동작한다
- 세션이 시작 시점의 버전을 고정해 읽는다
- Planner가 검증된 자료 결합 원칙에 따라 로드맵을 확장할 수 있다

## 10. 문서 변경 원칙

- 상위 요구사항 변경 전에 하위 계약 영향 범위를 먼저 확인한다
- 저장 구조 변경은 DB 물리 스키마 문서를 먼저 수정한다
- enum, validation, JSONB shape, state 변경은 계약 부록 문서를 먼저 수정한다
- API 필드 변경은 API 명세 정렬본을 먼저 수정한다
- 기능/화면 문서는 위 계약 변경이 반영된 뒤 후행 수정한다
