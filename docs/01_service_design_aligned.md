# AI 개발자 성장 코치 서비스 설계 원본 정렬본

## 1. 문서 목적 및 역할

이 문서는 AI 개발자 성장 코치 서비스의 상위 설계 기준 문서다. 기존 원본 설계를 제안서, 아키텍처 문서, 계약 문서 기준에 맞춰 다시 정렬한 버전이다.

이 문서의 역할
- 서비스의 범위와 단계 전략을 정의한다
- v1과 v2의 책임 경계를 설명한다
- 파생 문서 간 우선순위와 참조 기준을 고정한다
- 팀 개발 시 공통 해석의 기준점을 제공한다

이 문서는 상세 DDL, enum 전문, JSONB shape 전문, API 전문을 직접 반복하지 않는다. 해당 내용은 하위 계약 문서를 최종 기준으로 참조한다.

---

## 2. 문서 우선순위

문서가 충돌할 경우 우선순위는 아래 순서를 따른다.

1. 서비스 범위와 단계 전략: 이 문서
2. 요구사항: 요구사항 정렬본
3. 저장 구조: DB 물리 스키마 문서
4. enum, JSONB shape, validation, 상태 전이: 계약 부록 문서
5. 외부 인터페이스: API 명세 정렬본
6. 기능별 처리 책임: 기능 명세 정렬본
7. 화면 입력과 흐름: 사용자 흐름 및 화면 설계 정렬본

핵심 원칙
- 상위 문서는 방향과 범위를 정한다
- 하위 문서는 구현 계약을 닫는다
- 구현 충돌이 발생하면 더 구체적인 계약 문서를 우선한다

## 3. 서비스 개요

### 3.1 서비스명

AI 개발자 성장 코치

### 3.2 한 줄 소개

개발자의 현재 역량을 진단하고, GitHub 기반 실제 경험과 학습 우선순위를 연결해, 이후에는 AI 코치로 실행까지 이어주는 성장 지원 서비스다.

### 3.3 해결하려는 문제

- 자신의 현재 역량을 객관적으로 파악하기 어렵다
- 목표 직무 대비 부족한 기술을 구조적으로 알기 어렵다
- GitHub 경험, 학습 계획, 포트폴리오 정리가 분리되어 관리된다
- 무엇을 먼저 공부해야 할지 우선순위를 잡기 어렵다
- 계획을 세워도 행동으로 이어지지 않는 경우가 많다
- 학습 경험을 취업 준비와 포트폴리오 문서로 연결하기 어렵다

### 3.4 서비스가 제공하는 가치

이 서비스는 아래 흐름을 하나로 연결한다.

- 현재 상태 진단
- GitHub 기반 실제 경험 보정
- 부족 기술과 보완 포인트 파악
- 학습 우선순위 추천
- 실행 가능한 학습 로드맵 생성
- 포트폴리오 연결 가능성 강화
- 이후 AI 코치 기반 실행 지원

즉, 단순 추천 도구가 아니라 성장 과정을 구조화하고 다음 행동으로 이어주는 서비스다.

## 4. 타깃 사용자

### 4.1 주요 사용자

- 백엔드 개발자 취업 준비생
- 데브코스, 부트캠프, 강의 수강생
- 프로젝트 경험은 있지만 기술 깊이를 정리하기 어려운 주니어 개발자
- GitHub 경험을 취업 준비와 포트폴리오로 연결하고 싶은 사용자

### 4.2 사용자 목표

- 현재 역량 수준 파악
- 부족한 기술 확인
- 목표 직무 기준 학습 우선순위 정리
- GitHub 기반 실제 사용 기술 확인
- 학습 경험을 포트폴리오와 프로젝트 기술서로 연결
- 주차별 학습 로드맵 수립
- 장기적으로는 AI 코치와 함께 학습 실행 유지

## 5. 단계 전략

### 5.1 v1 전략

v1은 페이지 기반 기능 MVP다. 사용자가 입력하고 결과를 확인하는 흐름을 먼저 완성하는 것이 목적이다.

v1의 핵심 범위
- 프로필 저장
- GitHub OAuth 연동
- 저장소 선택과 GitHub 분석
- GitHub 결과 보정
- 역량 진단과 부족 스택 추천
- 학습 로드맵 생성
- 진도 체크와 결과 재조회

즉, v1은 단순 진단 페이지가 아니라 입력부터 결과 저장과 재조회까지 포함한 기능 MVP다.

### 5.2 v2 전략

v2는 v1 기능이 안정적으로 동작한 뒤, 그 결과를 재사용해 멀티 에이전트 구조를 얹는 확장 단계다.

v2의 핵심 확장
- Analyzer, Planner, Coach 역할 분리
- Context Manager 기반 상태 관리
- 이벤트 기반 피드백 루프
- Pattern Detector 기반 재분석 또는 재계획
- 일일 코치 대화
- Function Calling 기반 검증 자료 결합
- 코딩테스트 분석 draft

### 5.3 단계 분리 이유

- v1 MVP 범위가 멀티 에이전트 구조에 묻히지 않게 하기 위해
- 현재 구현 범위와 이후 확장 범위를 문서에서 명확히 구분하기 위해
- 파생 문서들이 같은 단계 기준을 공유하게 하기 위해
- v1 결과물을 v2에서 그대로 재사용하는 연결 구조를 분명히 하기 위해

## 6. 핵심 설계 원칙

### 6.1 인증과 사용자 식별

- 로그인 방식은 이메일 로그인과 OAuth 로그인을 함께 고려한다
- 보호 API의 사용자 식별은 JWT 인증 컨텍스트를 기준으로 처리한다
- 요청 body에 `userId`를 직접 싣는 방식은 사용하지 않는다

### 6.2 결과 이력 관리

- v1, v2는 서비스 단계 이름이다
- `version`은 개별 결과 이력 관리용이다
- 역량 진단, GitHub 분석, 학습 로드맵은 재실행 시 새 결과 row를 만든다
- 기본 조회는 최신 결과를 반환하고, 필요 시 과거 결과도 조회할 수 있다

### 6.3 로드맵 원본 기준

- 로드맵 원본은 `learning_roadmaps + roadmap_weeks + progress_logs`다
- `roadmap_payload`는 결과 보관 및 snapshot 조립용이다
- 화면 진도 상태는 `progress_logs` 최신 row 기준으로 계산한다

### 6.4 공통 계약 우선

- enum 값은 계약 부록 문서를 따른다
- JSONB는 자유 구조가 아니라 합의된 key shape를 따른다
- API 필드명, 필수/선택 여부, 에러 코드는 API 명세 정렬본을 따른다
- 화면 표시용 한글 라벨은 프론트에서 매핑하고 저장/전송 값은 계약 원문을 사용한다

### 6.5 v1 처리 방식

- v1 외부 계약은 기본 동기 처리로 시작한다
- 화면은 로딩 상태를 가질 수 있지만, 기본적으로 요청 완료 시 결과를 받는 흐름을 사용한다
- GitHub 분석과 장시간 자료 탐색은 이후 비동기 상태 모델로 확장할 수 있다

### 6.6 v2 확장 방식

- v2는 v1 페이지 기능을 대체하지 않는다
- v2는 v1 결과를 읽어 상위 경험 레이어를 제공한다
- 코치 세션은 시작 시점의 `profileVersion`, `roadmapVersion`을 고정한다
- Context Manager는 인프라 레이어이고, 비즈니스 판단은 에이전트가 담당한다

## 7. 핵심 사용자 흐름

### 7.1 v1 전체 흐름

1. 사용자가 로그인한다
2. 사용자가 프로필과 선택 자료를 입력하고 저장한다
3. 사용자가 GitHub OAuth 연동을 완료한다
4. 사용자가 분석 대상 저장소와 핵심 repo를 선택한다
5. 시스템이 전체 GitHub 정적 분석과 핵심 repo 요약을 수행한다
6. 사용자가 GitHub 분석 결과를 검토하고 필요 시 보정한다
7. 시스템이 프로필과 GitHub 최종 분석 결과를 기준으로 역량 진단을 수행한다
8. 시스템이 진단 결과를 바탕으로 학습 로드맵을 생성한다
9. 사용자는 주차별 계획을 확인하고 진도를 체크한다
10. 사용자는 대시보드에서 최근 결과를 다시 조회한다

### 7.2 v2 확장 흐름

v1에는 Coach 대화 인터페이스가 없다. Coach는 v2에서 추가되는 컴포넌트다.

1. v1에서 이미 로드맵이 생성된 사용자가 AI 코치 채팅방에 진입한다
2. 시스템이 코치 세션을 생성한다
3. 세션 생성 시점의 `profileVersion`, `roadmapVersion`을 고정한다
4. Coach가 Tier 기반 컨텍스트를 읽는다
5. 사용자가 메시지를 보내면 시스템이 응답과 의도를 반환한다
6. 필요 시 재분석 또는 재계획 이벤트가 발생한다
7. 코딩테스트 draft 기능은 별도 후순위 파이프라인으로만 확장한다

## 9. 데이터와 저장 구조 요약

### 9.1 v1 저장 구조

v1은 원본 입력과 진행 상태를 정규화된 구조로 유지하고, 분석 결과는 JSONB로 저장한다.

핵심 구조
- 사용자와 프로필
- 직무 기준표와 사용자 기술 스택
- GitHub 연결, 저장소 정보, GitHub 분석 결과
- 역량 진단 결과
- 학습 로드맵, 주차별 계획, 진도 로그

요약 원칙
- 입력과 원본 진행 상태는 정규화한다
- 분석 결과는 JSONB로 저장한다
- 결과 이력은 `version`으로 관리한다
- 로드맵 원본은 주차 테이블과 진도 로그 구조를 유지한다

### 9.2 v2 확장 구조

v2는 v1 결과를 바탕으로 snapshot, 세션, 이벤트, 패턴 감지 구조를 추가한다.

핵심 컴포넌트
- UserContextSnapshot
- ChatSession
- CoachConversation
- AgentEvent
- DetectedPattern

추가 원칙
- 코딩테스트 관련 데이터 구조는 v2 후순위 draft로만 남긴다
- snapshot은 원본 데이터를 대체하지 않는다

## 10. 역할 분담 및 오너십 원칙

### 10.1 역할 분담 기준

- v1 역할 분담 기준: 페이지 / 기능축
- v2 역할 분담 기준: 아키텍처 / 상태 관리 오너십 축

### 10.2 공통 계약 오너십

기능 오너와 별도로 아래 계약은 공통 오너가 최종 승인한다.

- DB 물리 스키마
- enum / code 사전
- JSONB shape
- validation 규칙
- 상태 전이 규칙
- API DTO와 에러 코드

이 원칙을 두지 않으면 공통 DTO, 상태, 저장 규칙이 병렬 개발 중 쉽게 어긋난다.

## 11. 파생 문서 기준

이 문서를 기준으로 아래 문서를 유지한다.

- 요구사항 정렬본: 범위와 기능 요구사항
- 사용자 흐름 및 화면 설계 정렬본: 입력과 이동 흐름
- 기능 명세 정렬본: 기능별 입력, 처리, 저장 규칙
- DB 물리 스키마 문서: 실제 저장 구조
- 계약 부록 문서: enum, JSONB shape, validation, state transition
- API 명세 정렬본: 외부 계약
- 아키텍처 문서: 구조와 컴포넌트 책임

공통 원칙
- 상위 문서의 방향과 하위 문서의 계약이 충돌하지 않아야 한다
- 구현 변경 시에는 가장 먼저 영향을 받는 계약 문서를 선반영한다
- 파생 문서는 한 문서만 바꾸는 방식이 아니라 관련 문서를 함께 갱신한다

## 12. 최종 정리

이 서비스는 먼저 페이지 기반 기능 서비스로 시작한다.

v1에서는 아래 기능을 핵심 MVP로 완성한다.
- 프로필 저장
- GitHub OAuth 연동과 저장소 선택
- GitHub 분석과 사용자 보정
- 역량 진단 + 부족 스택 추천
- 학습 로드맵 생성과 진도 체크

이후 v2에서는 위 결과를 바탕으로 멀티 에이전트 기반 AI 코치 구조를 확장한다.

끝까지 유지해야 하는 기준
- 인증은 JWT/OAuth 기준을 따른다
- 결과 이력은 `version`으로 관리한다
- 로드맵 원본은 `learning_roadmaps + roadmap_weeks + progress_logs`다
- enum, JSONB shape, validation, API 계약은 하위 문서 기준으로 닫는다
- v2는 v1을 대체하지 않고 그 위에 얹는 구조로 확장한다
